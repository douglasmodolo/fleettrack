// O Projeto

Sistema de Logística de Entregas

Simula a infraestrutura de entregas de um grande marketplace: pedidos entram, são roteados para entregadores, rastreados em tempo real e o cliente recebe notificações. Cada responsabilidade vira um microservice independente — exatamente como funciona em empresas de escala.

Conceitos técnicos aplicados
Por que MongoDB no Tracking e não Postgres?
Histórico de posições GPS é uma sequência de documentos com estrutura variável — sem joins, volume alto de escrita e schema que pode evoluir sem migrations. Document model é natural para isso.
Como funciona a idempotência no Order Service?
O cliente envia um idempotency-key no header. O service verifica no Redis se aquela chave já foi processada — se sim, retorna o resultado anterior sem reprocessar. Evita pedidos duplicados por retry ou clique duplo.
O que é Saga Choreography e onde está aqui?
Cada service reage a eventos sem um coordenador central. Order publica "order.created" → Tracking cria o rastreamento → Notification dispara aviso. Consistência eventual sem transação distribuída.
O que é Circuit Breaker e por que está no Gateway?
Evita cascade failure. Se o Tracking Service estiver lento, o Gateway para de chamar ele por um tempo e retorna um fallback — protege todos os outros services de sofrerem junto.
Como funciona o Optimistic Locking no JPA?
A entidade tem um campo @Version. Quando duas transações tentam atualizar o mesmo registro, a segunda recebe uma exceção — o banco não precisa de lock pesado. Ideal para baixa contenção.
Por que Testcontainers nos testes de integração?
Sobe instâncias reais de Postgres, MongoDB e Redis via Docker durante o teste — sem mocks de banco. O teste valida o comportamento real, não uma simulação. Derruba tudo sozinho ao terminar.
O que acontece se o Kafka cair?
O Outbox Pattern garante que eventos não se percam: o Order Service grava o evento na própria tabela do Postgres na mesma transação do pedido. Um worker publica no Kafka depois — atomicidade garantida.
Como o traceId conecta logs de services diferentes?
O Micrometer Tracing gera um traceId na entrada do Gateway e o propaga via header HTTP e Kafka headers. O MDC injeta esse ID em todos os logs automaticamente — dá pra rastrear uma requisição de ponta a ponta.
